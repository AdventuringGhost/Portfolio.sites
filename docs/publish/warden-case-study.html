What it does

Threat detection to auto-remediation in one pipeline

Warden is a Python FastAPI webhook server that sits between your Kubernetes security tooling and your on-call engineer. It receives alerts from two sources, classifies them with an AI model, and acts — so routine incidents resolve themselves and humans only get paged when it genuinely matters.

01

Receive alerts

Falco streams runtime security events (shell spawns, privilege escalation, suspicious syscalls) via Falcosidekick to Warden's /webhook endpoint. OPA Gatekeeper admission violations feed the same pipeline.

02

Triage with Claude Sonnet

Warden sends the structured alert payload to Claude Sonnet 4.6 with a system prompt that defines severity rubrics and response policy. Claude returns a severity classification (low / medium / high) and a structured remediation recommendation.

03

Act on severity

Low severity — Warden auto-patches the affected workload immediately (pod restart, label quarantine, network policy application) and logs the action.

High severity — Warden drafts a plain-language remediation runbook and surfaces it for human review. No automated action is taken until a human approves.

04

Observe and audit

All triage decisions and patch actions are emitted as Prometheus metrics. Every AI decision is logged with the original alert payload, Claude's reasoning, the action taken, and a timestamp — giving a complete audit trail for post-incident review.

Architecture

Two-layer security on AKS

Warden enforces security at two distinct points in the Kubernetes lifecycle: before a workload is admitted to the cluster, and while it is running. Together they cover the full threat surface — misconfigurations that would never even reach production, and adversarial behaviour that only appears at runtime.

warden · architecture overview
🛡️
Layer 1 — OPA Gatekeeper (Admission)

Intercepts every API call to the Kubernetes control plane before the object is persisted. Enforces ConstraintTemplates written in Rego: blocks privileged containers, restricts image registries to verified sources, rejects non-root violations. If the manifest fails a constraint, the pod never reaches a node.

↓ workloads that pass admission
👁️
Layer 2 — Falco (Runtime)

Runs as a privileged DaemonSet on every node. Uses eBPF kernel probes to watch every syscall in real time. Detects shell spawns inside containers, privilege escalation attempts, unexpected network connections, and file system tampering. Events are forwarded to Warden via Falcosidekick.

↓ events forwarded to webhook
🤖
Warden — FastAPI Webhook + Claude Sonnet

Receives the alert, structures it into a prompt, and queries Claude Sonnet 4.6 for triage. Claude returns severity + remediation recommendation. Warden applies the decision: auto-patch for low-severity, runbook draft for high-severity. API keys stored in Azure Key Vault, injected at runtime.

↓ metrics emitted
📊
Observability — Prometheus

Custom metrics track alert volume by severity, auto-patch success/failure rate, Claude API latency, and runbook generation count. Scraped by the cluster's Prometheus instance and queryable for post-incident analysis.

Challenges & fixes

Real problems. Root causes. Actual fixes.

Every one of these was a real blocker hit during development on a live cluster. The fixes are in the repo. The lessons are below.

01
OPA blocked Falco's own DaemonSet
Problem

The AllowedImageRegistries and BlockPrivilegedContainers constraints were cluster-wide. Falco itself runs as a privileged container pulling from Docker Hub — so OPA denied its own DaemonSet pods before they could start. Falco was silently blocked by the security layer it was supposed to complement.

Fix

Added a namespace exclusion to both ConstraintTemplates: the falco namespace is exempted from the privileged container and registry constraints. Constraints still enforce everywhere else.

Lesson

Security tooling often needs elevated permissions to do its job. Namespace-scoped exclusions are the right pattern — blanket constraint relaxation is not.

02
WSL2 eBPF limitations blocked local Falco detection
Problem

Falco's eBPF probe requires access to kernel tracepoints. WSL2 runs a modified kernel that strips many of these, so local Falco test runs produced no events — making it impossible to validate the pipeline locally before deploying to AKS.

Fix

Documented the WSL2 limitation clearly and moved end-to-end pipeline validation to AKS from the outset. Local dev used mocked webhook payloads for Warden unit testing; real eBPF detection was only validated on-cluster.

Lesson

Know which parts of your stack require real hardware. Design your local dev loop around that constraint rather than fighting it.

03
AllowedImageRegistries schema validation errors
Problem

The parameterised ConstraintTemplate for AllowedImageRegistries used a CRD schema with an openAPIV3Schema for the parameters.registries field. OPA Gatekeeper rejected the schema on certain AKS versions due to validation webhook conflicts between the CRD admission controller and Gatekeeper's own webhook.

Fix

Removed the parameterised schema entirely. Hardcoded the allowed registries directly into the Rego policy. Fewer moving parts, no schema validation surface, and the policy is easier to audit at a glance.

Lesson

Parameterised OPA constraints add flexibility but also a schema validation layer that can fail unpredictably across Kubernetes versions. For a small known-registry list, inline Rego is simpler and more reliable.

04
Agent code not present in the container image
Problem

The Warden webhook server needed to run inside the cluster but publishing a private container image for every dev iteration was too slow. The agent code was not baked into any available image.

Fix

Mounted the agent source code as a Kubernetes ConfigMap and ran it via a base Python image. The pod pulls the code from the ConfigMap at startup and executes it — no custom image build required for development iterations.

Lesson

ConfigMap-mounted code is a legitimate pattern for rapid iteration on cluster-internal tooling. It trades image immutability for development speed — acceptable for a dev/staging context, not for production.

05
Corrupted API key in environment variable
Problem

The Claude API key injected from Azure Key Vault into the pod's environment was being silently corrupted — extra whitespace or encoding artefacts caused every Claude API call to return 401 Unauthorized. The key looked valid in the logs but was not.

Fix

Detected the corruption, rotated the key immediately, and added an explicit strip-and-validate step in the Warden startup code that reads the key, strips whitespace, checks length, and fails fast with a clear error if the key does not look valid before accepting any traffic.

Lesson

Secrets injected from external vaults can be corrupted in transit. Always validate secrets at application startup, rotate immediately on detection, and treat 401s from downstream APIs as a secret-validity signal, not just an auth error.

Retrospective

What I'd do differently

Honest engineering means knowing what the next iteration looks like.

→

Build the container image from the start. The ConfigMap pattern was useful for rapid iteration but should be replaced with a proper CI pipeline that builds and pushes a versioned image on every commit. The Azure DevOps pipeline is already set up — this is the natural next step.

→

Add human approval UX for high-severity runbooks. Right now, runbooks are drafted and logged. A production version needs a way to surface them — a Slack notification, a GitHub issue, a webhook to a ticketing system — so human review actually happens in a real workflow, not just in the logs.

→

Test Falco rule tuning before going to AKS. Some detection rules were too broad initially and generated noise. With WSL2 limitations documented upfront, the right move is to invest earlier in local mock payload testing and rule refinement before spending cluster time chasing false positives.

→

Secret validation on every startup, not just after a 401. The corrupted API key was caught reactively. A startup health check that validates secret format and makes a test API call before marking the pod Ready would catch this class of issue before any real traffic arrives.
