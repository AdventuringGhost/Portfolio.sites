The AI Doesn't Get to Decide Who Sees What

What it does

Policy-enforced AI in four components

Covenant is a FastAPI service that layers four components in strict order. Each handles exactly one responsibility. The sequence is the security model: identity first, policy second, retrieval third, generation last.

🪪
JWT

Identity

Every request carries a Bearer token. FastAPI middleware validates the signature, extracts role, user_id, and tenant_id, and builds the claims dict passed to OPA.

🛡️
OPA

Policy gate

OPA evaluates covenant.authz.allow against the JWT claims, request path, and body. If the result is false, FastAPI returns 403 immediately — Claude never runs.

🔍
pgvector

Scoped retrieval

The query is embedded and run as a cosine similarity search over the documents table, filtered by tenant_id. The AI only sees documents it is already permitted to see.

🤖
Claude

Generation

Claude Sonnet 4.6 generates a response from the permitted documents. The system prompt is cached via cache_control: ephemeral. Claude never touches a request OPA denied.

Three-role RBAC proof

Who can see what — enforced by Rego

Covenant ships with three roles. The access matrix is defined entirely in Rego. Changing a permission means changing a policy file and committing a PR — not touching application code.

Endpoint admin user auditor
POST /query (regular) ✓ 200 ✓ 200 ✗ 403
POST /query (sensitive: true) ✓ 200 ✗ 403 ✗ 403
POST /query (cross-tenant) ✓ 200 ✗ 403 ✗ 403
GET /audit ✓ 200 ✗ 403 ✓ 200

The same request — POST /query with {"sensitive": true} — hits OPA with a different JWT each time. The response is determined entirely by the Rego policy. 

# admin JWT: role=admin, tenant_id=acme
curl -s -X POST http://localhost:8000/query \
  -H "Authorization: Bearer <admin-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"query": "show Q4 revenue breakdown", "tenant_id": "acme", "sensitive": true}'

# OPA input: {role: "admin", path: "/query", body.sensitive: true}
# OPA result: allow = true  (admin rule matches)

HTTP 200 OK
{
  "response": "Q4 revenue: $4.2M total, up 18% YoY. Breakdown by region...",
  "documents_used": 3,
  "opa_decision": "allow"
}
# user JWT: role=user, tenant_id=acme
curl -s -X POST http://localhost:8000/query \
  -H "Authorization: Bearer <user-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"query": "show Q4 revenue breakdown", "tenant_id": "acme", "sensitive": true}'

# OPA input: {role: "user", path: "/query", body.sensitive: true}
# OPA result: allow = false  (user rule requires not input.body.sensitive)

HTTP 403 Forbidden
{
  "error": "access denied",
  "policy": "user:sensitive-blocked",
  "opa_decision": "deny"
}
# auditor JWT: role=auditor, tenant_id=acme
curl -s -X POST http://localhost:8000/query \
  -H "Authorization: Bearer <auditor-jwt>" \
  -H "Content-Type: application/json" \
  -d '{"query": "show Q4 revenue breakdown", "tenant_id": "acme", "sensitive": true}'

# OPA input: {role: "auditor", path: "/query", body.sensitive: true}
# OPA result: allow = false  (auditor has no /query rule — default deny)

HTTP 403 Forbidden
{
  "error": "access denied",
  "policy": "auditor:query-denied",
  "opa_decision": "deny"
}
policy/covenant.authz — the complete access policy Rego v1 
package covenant.authz

import rego.v1

# Default to deny — nothing passes unless explicitly permitted
default allow := false

# admins can query everything, including sensitive and cross-tenant
allow if {
    input.claims.role == "admin"
    input.path in {"/query", "/audit"}
}

# users can query non-sensitive docs within their own tenant only
allow if {
    input.claims.role == "user"
    input.path == "/query"
    not input.body.sensitive
    input.claims.tenant_id == input.body.tenant_id
}

# auditors can read the audit log — and nothing else
allow if {
    input.claims.role == "auditor"
    input.path == "/audit"
}

Architecture

Six steps — OPA is the gate

The request flow is linear. Each step hands off to the next only if the previous step passed. The OPA check is not advisory — it is a hard stop in the middleware chain. Nothing downstream runs if Rego returns false.

01

Client sends request with Bearer JWT

The client POSTs to /query with a JSON body and an Authorization: Bearer <jwt> header. No other auth mechanism is accepted.

02

JWT middleware validates and extracts claims

FastAPI middleware verifies the JWT signature against the HS256 secret, checks expiry, and extracts role, user_id, and tenant_id into a claims dict. A missing or invalid token returns 401 before OPA is ever called.

03
🛡️ OPA access gate

Policy check — hard ALLOW or DENY

FastAPI POSTs the input bundle {claims, path, method, body} to the OPA REST API at http://opa:8181/v1/data/covenant/authz. OPA evaluates covenant.authz.allow. If the result is true, execution continues. If false, FastAPI returns 403 Forbidden immediately. Claude is never called.

04

pgvector semantic search — scoped to tenant

The query text is embedded via sentence-transformers and run as a cosine similarity search against the documents table, filtered by tenant_id = claims.tenant_id. Cross-tenant results are impossible at the query level — not just excluded from the response.

05

Claude generates response from permitted documents

The permitted documents are passed to Claude Sonnet 4.6 alongside the user query. The system prompt is cached with cache_control: ephemeral, keeping latency low on repeated requests. Claude sees only what OPA allowed through.

06

Response returned + audit log written

The response is returned to the client. An audit log entry is written with timestamp, user_id, role, tenant_id, query_hash, docs_returned, and opa_decision. Every access — allowed or denied — is on the record.

Challenges & fixes

Real blockers. Root causes. Actual fixes.

Every one of these was a genuine development blocker. The solutions are in the repo.

01
OPA Docker image ran a REST server, not an eval binary
Problem

The official openpolicyagent/opa Docker image starts a REST server on port 8181. The code was calling OPA as if it were a CLI eval tool, passing policy files as arguments — which works locally but silently fails inside the container because the server ignores CLI-style args and just listens for HTTP requests.

Fix

Committed to the REST server model entirely. Docker Compose starts OPA with opa run --server --addr :8181 /policies, and FastAPI sends policy input via POST /v1/data/covenant/authz. Policies are mounted as a volume so edits apply without a rebuild.

Lesson

OPA has three deployment modes: sidecar REST server, Go library (in-process), and CLI eval. Pick one before you write a line of integration code. Mixing mental models wastes hours.

02
Rego v1 syntax broke existing policy files
Problem

OPA 0.68+ requires import rego.v1 or the --v1-compatible flag. Legacy Rego using = instead of := in rule heads throws a cryptic parse error that doesn't say "version mismatch" — it just says the rule is invalid.

Fix

Added import rego.v1 to every policy file. Pinned openpolicyagent/opa:0.68.0 in docker-compose.yml so future pulls don't silently break policies. Rego v1 is stricter and clearer — the if keyword in rule heads reads like intent.

Lesson

Pin your OPA version. Rego v1 is the right direction, but the error messages don't always point at the version as the root cause.

03
pgvector requires superuser to install the extension
Problem

CREATE EXTENSION vector requires superuser or the pg_extension_owner role. The application DB user covenant_app can't run it. The Docker Compose init script ran as the app user by default, so the extension was never created — causing a silent column type "vector" does not exist error at migration time.

Fix

Split docker-entrypoint-initdb.d/ into two files: 00_superuser.sql (runs as postgres, creates the extension) and 01_app.sql (runs as covenant_app, creates schema and tables). PostgreSQL processes init scripts alphabetically — numeric prefixes enforce ordering.

Lesson

DB initialisation ordering is load-bearing. Name init scripts with numeric prefixes. Know which operations need superuser before you write the scripts.

04
pgvector Docker build failed on WSL2 — missing kernel headers
Problem

Attempting to build pgvector from source inside a Docker container on WSL2 failed because pgvector's Makefile links against PostgreSQL server headers that require matching kernel build tools. The WSL2 kernel doesn't ship full headers, so make exited with a cryptic pg_config: not found error even when PostgreSQL was installed.

Fix

Switched from building pgvector from source to using the official pgvector/pgvector:pg16 image, which ships pre-compiled binaries. No kernel build step, no WSL2 header mismatch, extension ready on container start.

Lesson

Don't build database extensions from source unless you need a specific unreleased version. The official prebuilt image exists for a reason.

05
Short-lived JWT expiry caused random 401s during demos
Problem

Test JWTs used a 15-minute expiry — which is a sensible production default but causes random 401s during extended demo sessions. The error surfaced as an OPA claim-validation failure, not a clear "token expired" message, making it confusing to debug mid-demo.

Fix

Added a get_token() helper that inspects the exp claim before every request and auto-refreshes if the token is within 60 seconds of expiry. Demo tokens now use a 24-hour TTL configurable via .env — kept separate from the production token lifetime setting.

Lesson

Separate "demo token TTL" from "production token TTL" in config from day one. Auth debugging during a live demo is the worst kind of debugging.

What this demonstrates

Skills proven end-to-end

📜

Policy-as-infrastructure

OPA Rego policies live in version control. Access rules are reviewable PRs, tested in CI, and deployed like code. Changing a permission is a diff, not a database row or a config file nobody owns.

🚫

Default-deny

default allow := false — every identity starts with zero access. Permissions are explicit grants. Nothing passes unless a Rego rule explicitly permits it.

📋

Audit trail

Every OPA decision is written to the audit log with the full input bundle — who asked, what role, what path, what was allowed or denied. Auditors can query the log; they cannot query documents.

🛡️

Hard gate — AI cannot bypass access control

OPA evaluates in the middleware chain, before Claude runs. There is no system prompt that bypasses Rego. A 403 from OPA is a 403 — the model never sees the request, so it cannot be prompted around the policy.

🔍

Auditor isolation

Auditors have an explicit ALLOW on /audit and an explicit DENY on /query. This is not an oversight in the policy — it is the policy. The separation is proven by the RBAC demo.

⚡

Prompt caching

Claude's system prompt uses cache_control: ephemeral. The cache TTL is 5 minutes — repeated requests in the same session hit the cache and return significantly faster. Near-zero latency penalty on high-frequency query endpoints.

Industry use cases

The same architecture, applied to real compliance problems

Three industries where policy-enforced AI directly addresses regulatory and operational risk.

💳
Fintech

Multi-tenant financial report access

  • Analysts query only their assigned client's data — tenant isolation enforced at the pgvector level
  • Admins access cross-tenant reports; OPA rule explicit, auditable, reviewed in PRs
  • Auditors read the full decision log for compliance evidence without touching financial records
  • Rego policy change for a new data class takes minutes, not a sprint
🏥
Healthtech

HIPAA-adjacent patient record isolation

  • Clinicians query only their assigned patient cohort — cross-patient data blocked at OPA before Claude sees it
  • AI cannot be prompted to surface another patient's records — the gate is in the middleware, not the model
  • Audit log provides access evidence for HIPAA review without querying the records themselves
  • Role changes take effect on next request — no cache poisoning, no stale session state
🏭
Manufacturing

IP-sensitive technical document search

  • Product line engineers query only their product's documents — pgvector filter enforces isolation
  • Auditors read change logs for their assigned product lines; they cannot query design specs
  • New product line access provisioned by adding a Rego rule, not a DB role grant
  • IP leakage between product lines is impossible at the retrieval layer
